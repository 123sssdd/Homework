# custom-power

定制项目全流程 skill 集合，覆盖从信息收集到成果物下载的完整链路。

## 流程总览

```
收集信息 → 创建版本 → 拉取代码 → AI开发 → 生成报告 → 正式构建
```

## Skills

| Skill | 说明 | 使用文档 |
|-------|------|----------|
| custom-info-collector | 从 highway/用户输入收集定制项目信息 | [查看文档](./docs/custom-info-collector.md) |
| hido-create-version | browser-harness 自动在 hido 创建定制版本 | — |
| custom-code-setup | 拉取代码并初始化开发目录 | [查看文档](./docs/custom-code-setup.md) |
| custom-develop | 编排 prompts 工作流驱动 AI 定制开发（含自测） | [查看文档](./docs/custom-develop.md) |
| test-report-generator | 基于模板生成自测报告 | [查看文档](./docs/test-report-generator.md) |
| hido-build-release | hido 正式构建与成果物下载 | — |

## 使用方式

按流程顺序向 Claude 描述意图，Claude 自动匹配对应 skill：

1. **收集信息**：提供项目名、组件标识、JKN、highway 链接、需求描述
2. **创建版本**：确认信息后自动触发 hido 版本创建
3. **拉取代码**：收到邮件后粘贴代码地址
4. **开始开发**：自动走需求检查 → 文档生成 → 代码实现 → 自测
5. **生成报告**：自测通过后生成测试报告
6. **正式构建**：在 hido 点击正式构建并下载成果物

# 定制项目代码拉取与环境初始化

从用户输入或邮件获取代码仓库地址，自动识别 git/svn，拉取代码并创建标准化开发目录结构。

**本 skill 可独立使用**：只要用户提供了代码仓库地址（或提到拉代码），即可直接触发，不需要先运行 custom-info-collector。

## 前置条件

- 用户提供了代码仓库地址，或提到要拉代码/clone/checkout
- 本机已安装对应的版本控制工具（git 或 svn）

## 可选前置（非必需）

- custom-info-collector 已完成：可自动获取 `component`、`projectName` 等信息
- hido 版本已创建：邮件通知包含代码地址

## 参数

| 参数 | 变量名 | 必填 | 说明 |
|------|--------|------|------|
| 代码仓库地址 | `repoUrl` | 是 | git 或 svn 仓库地址，可从邮件获取 |
| 代码仓库类型 | `repo` | 否 | git/svn，默认从 custom-info-collector 获取，fallback 为 svn |
| 目标分支 | `branch` | 否 | git 分支名，默认 master/main |
| 工作目录 | `workDir` | 否 | 代码存放路径，默认为 `D:\定制项目\({component}){projectName}` |

## 执行步骤

### Step 1: 获取代码仓库地址

优先级：
1. 用户直接提供 `repoUrl`
2. 从邮件中提取（提示用户粘贴邮件中的代码地址）
3. 如果用户说"邮件还没到"，暂停并提示：

```
请等待 hido 邮件通知，收到后粘贴邮件中的代码仓库地址。
```

不要自动检查邮箱或轮询等待。

### Step 1.5: 判断代码仓库类型

**不要依赖用户手动指定类型**，根据 URL 特征自动判断。按以下优先级依次匹配：

**规则 1**：URL 以 `.git` 结尾 → **Git**
- 示例：`https://sys-gitlab.hikvision.com.cn/PBG/components/general/visitor.git`

**规则 2**：URL 为 `git@...` 或 `ssh://git@...` → **Git**

**规则 3**：`svn://` 开头 → **SVN**

**规则 4**：URL 路径中包含 `/trunk/`、`/branches/`、`/tags/` → **SVN**
- 示例：`https://192.0.0.241/USTA-visitor/branches/custom/IPM.../JKN.../visitor`
- 这是 SVN 的典型目录结构，**即使 URL 是 `https://` 开头也是 SVN，不是 Git**

**规则 5**：以上都不匹配 → 询问用户

**核心区分点**：两者都是 `https://` 开头，但 Git 地址以 `.git` 结尾，SVN 地址路径中间有 `/branches/` 等目录层级。

### Step 2: 确定工作目录

文件夹命名格式：`(组件标识)项目名称`，例如 `(visitor)访客端`、`(scan)扫码端`。

获取 `{component}` 和 `{projectName}` 的优先级：
1. 同一会话中已运行过 custom-info-collector，直接从对话上下文中获取
2. 读取当前目录下 `docs/requirement.md`，从中提取 component 和项目名称（跨会话场景）
3. 从 URL 中推断（如 `USTA-visitor/.../visitor` → component 为 `visitor`，projectName 为 `访客端`）
4. 以上都无法确定时，**不要给默认值**，直接询问用户文件夹名称

```
无法自动确定文件夹名称，请提供工作目录的文件夹名（格式如 (visitor)访客端）：
```

确定后确认：

```
工作目录将创建在：D:\定制项目\({component}){projectName}\
请确认，或提供自定义路径。
```

**注意**：每个 Claude Code 终端是独立会话，无法访问其他终端的对话上下文。跨会话传递信息的唯一方式是通过文件（如 `docs/requirement.md`）。

### Step 3: 拉取代码

#### 3a. SVN 拉取

SVN 的分支路径已包含在 URL 中（如 `/branches/custom/...`），直接 checkout 完整 URL。

SVN 需要认证时，先向用户询问用户名和密码，然后用非交互参数执行：

```
SVN 仓库需要认证，请提供用户名和密码。
```

```bash
svn checkout {repoUrl} "{workDir}\code" --username {username} --password {password} --non-interactive
```

如果认证失败，提示用户检查用户名密码并重新输入。

#### 3b. Git 拉取

```bash
git clone -b {branch} {repoUrl} "{workDir}\code"
```

Git 如果也需要认证，同样向用户询问用户名和密码（或 token），使用带凭证的 URL 执行：

```bash
git clone -b {branch} https://{username}:{password}@{repoUrl} "{workDir}\code"
```

#### 3c. 验证拉取结果

```bash
ls "{workDir}\code"
```

确认 `code/` 目录下有项目文件。如果拉取失败（网络、认证等），报告错误让用户处理。

### Step 4: 创建开发目录结构

在 `workDir` 下创建标准化目录：

```
{workDir}/
├── code/          # 项目代码（已拉取）
└── docs/          # 开发文档目录（新建）
    └── .gitkeep
```

```bash
mkdir "{workDir}\docs"
```

`docs/` 目录用于后续 `custom-develop` skill 生成开发文档：
- `docs/requirement.md` — 需求文档（如已由 custom-info-collector 生成，复制过来）
- `docs/spec.md` — 功能大纲
- `docs/tasks.md` — 任务列表
- `docs/test_steps.md` — 测试步骤
- `docs/checklist.md` — 验收清单

### Step 5: 复制需求文档

如果 `custom-info-collector` 已生成 `docs/requirement.md`，将其复制到新的工作目录：

```powershell
if (Test-Path "docs\requirement.md") {
    Copy-Item "docs\requirement.md" "{workDir}\docs\requirement.md"
}
```

### Step 6: 输出项目上下文

向用户确认环境已就绪：

```

开发文档目录：{workDir}\docs
需求文档：{workDir}\docs\requirement.md

环境准备完成，可以开始定制开发（custom-develop）。
```

## 注意事项

2. **认证问题**：svn/git 可能需要用户预先配置凭证，拉取失败时提示用户处理认证

5. **工作目录约定**：所有后续 skill（custom-develop、test-report-generator）都以 `workDir` 为工作目录执行

# AI 驱动定制开发

编排 prompts 目录下的 prompt 工作流，实现从需求到自测的完整定制开发流程。

## 前置条件

- 已完成 custom-code-setup（代码已拉取，目录结构就绪）
- `workDir` 下存在 `code/` 和 `docs/` 目录
- `docs/requirement.md` 已存在（由 custom-info-collector 生成）

## 工作目录约定

所有操作在 `workDir` 下执行：

```
{workDir}/
├── code/                  # 项目代码（只读参考 + 开发修改）
└── docs/
    ├── requirement.md     # 需求文档（输入）
    ├── spec.md            # 功能大纲（生成）
    ├── tasks.md           # 任务列表（生成）
    ├── test_steps.md      # 测试步骤（生成）
    └── checklist.md       # 验收清单（生成）
```

## 参数

| 参数 | 变量名 | 必填 | 说明 |
|------|--------|------|------|
| 工作目录 | `workDir` | 是 | 定制项目根目录 |
| 测试环境信息 | `envInfo` | 否 | 测试步骤执行时的环境信息（URL、账号等） |

## 流程总览

```
┌─────────────────────────────────────────────────────┐
│ Phase 1: 需求确认                                    │
│   requirement_check ──(不清晰)──→ refine_requirement │
│          │                        │                  │
│          │←────────(循环)─────────┘                  │
│          ↓(清晰)                                     │
│   用户确认需求 → [可调整] → ai_adjust_requirement    │
├─────────────────────────────────────────────────────┤
│ Phase 2: 文档生成                                    │
│   generate_docs → [spec, tasks, test_steps, checklist]│
│   用户审核 → [可调整] → ai_adjust_document           │
├─────────────────────────────────────────────────────┤
│ Phase 3: 代码实现                                    │
│   ai_develop (按 tasks.md 逐任务执行)                │
├─────────────────────────────────────────────────────┤
│ Phase 4: 自测                                       │
│   test (按 test_steps.md 执行浏览器测试)             │
│   ──(有失败)──→ adjust_development_tasks → ai_develop │
│          │                        ↑                  │
│          │←───────(循环)──────────┘                  │
│          ↓(全部通过)                                 │
│   输出自测报告 → 触发 test-report-generator           │
└─────────────────────────────────────────────────────┘
```

## 执行步骤

### Phase 1: 需求确认

#### 1a. 需求清晰度检查

使用 `prompts/requirement_check.md` 作为 prompt，读取 `docs/requirement.md` 和 `code/` 目录结构，判断需求是否足够清晰。

将 prompt 中的内容作为系统指令，读取工作目录下的对应文件执行分析。

输出 JSON：
```json
{ "is_clear": true/false, "missing_items": [...] }
```

#### 1b. 需求细化（如果不清晰）

如果 `is_clear` 为 false，使用 `prompts/refine_requirement.md` 作为 prompt：

- `{{requirement}}` ← `docs/requirement.md` 的内容
- `{{missing_items}}` ← 上一步的 `missing_items`
- `{{adjustment_opinion}}` ← 用户的补充说明

向用户展示不清晰的要点，请用户补充后重新执行。细化结果写回 `docs/requirement.md`。

重新执行 1a 检查，直到 `is_clear` 为 true。

#### 1c. 用户确认需求

需求清晰后，向用户展示最终需求文档，询问：
- 确认无误 → 进入 Phase 2
- 需要调整 → 使用 `prompts/ai_adjust_requirement.md` 修改，`{{adjustment_opinion}}` 为用户的调整意见，修改后重新确认

### Phase 2: 文档生成

#### 2a. 生成开发文档

使用 `prompts/generate_docs.md` 作为 prompt，读取 `docs/requirement.md` 和 `code/` 目录，按顺序生成：

1. `docs/spec.md` — 功能大纲
2. `docs/tasks.md` — 任务列表（每任务含 ID、描述、状态、依赖）
3. `docs/test_steps.md` — 测试步骤
4. `docs/checklist.md` — 验收清单

每个文档生成后立即写入文件。

#### 2b. 用户审核文档

向用户展示生成的四个文档摘要，询问：
- 确认无误 → 进入 Phase 3
- 需要调整 → 使用 `prompts/ai_adjust_document.md` 修改，`{{adjustment_opinion}}` 为用户意见，修改时保持四份文档一致性

### Phase 3: 代码实现

#### 3a. 逐任务开发

使用 `prompts/ai_develop.md` 作为 prompt：

1. 读取 `docs/tasks.md`，找到第一个 `pending`（`[ ]`）状态的任务
2. 读取 `docs/spec.md` 获取功能设计上下文
3. 在 `code/` 目录下实现该任务
4. 更新 `docs/tasks.md` 中该任务状态为 `completed`（`[x]`）
5. 输出 JSON：`commit_message`、`update_note`、`modified_files`
6. 重复直到所有任务完成

**异常恢复**：如果中途中断，重新执行时从第一个 pending 任务继续。

#### 3b. 用户确认实现

所有任务完成后，向用户展示 `commit_message` 和 `update_note`，确认实现内容。

### Phase 4: 自测

#### 4a. 执行测试

使用 `prompts/test.md` 作为 prompt：

- `{{test_steps}}` ← `docs/test_steps.md` 的内容
- `{{env_info}}` ← 用户提供的 `envInfo`，或询问用户提供测试环境信息

按测试步骤顺序执行，记录结果，输出格式为 `===TEST_REPORT_START===` ... `===TEST_REPORT_END===` 包裹的测试报告。

#### 4b. 测试结果处理

**全部通过**：
- 告知用户自测通过
- 提示触发 `test-report-generator` 生成正式自测报告

**有失败用例**：
1. 使用 `prompts/adjust_development_tasks.md` 调整任务清单：
   - `{{test_report}}` ← 测试报告
   - `{{current_tasks}}` ← `docs/tasks.md`
   - `{{code_structure}}` ← `code/` 目录结构
   - 输出调整后的任务列表，写回 `docs/tasks.md`
2. 回到 Phase 3a 重新执行新增/修改的任务
3. 重新执行 Phase 4a 测试
4. 循环直到全部通过，或用户决定手动处理

**循环次数限制**：测试-修复循环最多 3 轮。超过 3 轮仍有失败，暂停并报告用户，由用户决定是否继续或手动介入。

### 完成输出

全部通过后输出：

```
定制开发完成。
- 任务列表：{workDir}/docs/tasks.md（全部 completed）
- 测试报告：{workDir}/docs/test_report.md
- 代码目录：{workDir}/code/

下一步：
1. 触发 test-report-generator 生成正式自测报告
2. 自测通过后触发 hido-build-release 正式构建
```

## Prompt 使用规范

所有 prompt 文件位于 `{custom-power项目路径}/prompts/` 目录下，使用方式：

1. 读取 prompt 文件内容作为系统角色和指令
2. 将 prompt 中的 `{{变量}}` 替换为实际值
3. 在 `workDir` 下执行（确保 prompt 中的相对路径 `docs/`、`code/` 指向正确位置）
4. 严格遵循 prompt 指定的输出格式

| Prompt 文件 | 使用的变量 | 用途 |
|-------------|-----------|------|
| requirement_check.md | 无模板变量 | 检查 `docs/requirement.md` 清晰度 |
| refine_requirement.md | `{{requirement}}`, `{{missing_items}}`, `{{adjustment_opinion}}` | 细化不清晰的需求 |
| ai_adjust_requirement.md | `{{adjustment_opinion}}` | 用户调整需求文档 |
| generate_docs.md | 无模板变量 | 生成 spec/tasks/test_steps/checklist |
| ai_adjust_document.md | `{{adjustment_opinion}}` | 用户调整开发文档 |
| ai_develop.md | 无模板变量 | 逐任务实现代码 |
| test.md | `{{test_steps}}`, `{{environment_info}}` | 执行自动化测试 |
| adjust_development_tasks.md | `{{test_report}}`, `{{current_tasks}}`, `{{code_structure}}` | 根据测试失败调整任务 |

## 注意事项

1. **工作目录切换**：每个 Phase 执行前确认当前工作目录为 `workDir`，prompt 中的 `docs/` 和 `code/` 相对路径才能正确解析
2. **任务状态持久化**：`docs/tasks.md` 中的任务状态是断点恢复的依据，每完成一个任务必须立即更新
3. **测试-修复循环上限**：最多 3 轮，避免无限循环。超过后暂停交由用户决定
4. **用户介入点**：Phase 1 确认需求、Phase 2 审核文档、Phase 3 确认实现 — 这三个节点必须等用户确认后才能继续
5. **prompt 文件只读**：不修改 `prompts/` 目录下的任何文件，只读取内容作为指令
6. **output format 严格遵循**：`requirement_check` 输出 JSON、`ai_develop` 输出 JSON、`test` 输出 `===TEST_REPORT===` 包裹格式，不要自定义格式

description: 收集并结构化定制项目信息，从 highway 需求评估单、JKN 工单或用户输入中提取项目名、组件标识、需求描述等关键信息，供后续 skill（hido-create-version、custom-develop、test-report-generator）使用。触发场景：(1) 用户提到"定制项目"、"收集信息"、"拿到定制信息" (2) 开始定制流程的第一步 (3) 用户提供 highway 链接或 JKN 单号 (4) 用户提到"需求评估单"。

# 定制项目信息收集

从 highway 需求评估单、JKN 工单或用户直接输入中收集定制项目信息，结构化存储供后续 skill 消费。

## 需要收集的信息

| 信息项 | 变量名 | 必填 | 来源 | 说明 |
|--------|--------|------|------|------|
| 项目名称 | `projectName` | 是 | highway/用户输入 | 完整项目名称 |
| 组件名称/标识 | `component` | 是 | highway/用户输入 | hido 平台上的组件标识 |
| JKN 单号 | `jkn` | 是 | highway 页面"订单编号" | 定制订单编号，格式如 `JKN20260506_952` |
| 需求评估单链接 | `requirementUrl` | 是 | 用户输入 | highway 需求评估单 URL |
| 需求描述 | `requirements` | 是 | highway 页面/用户输入 | 定制需求的具体内容，可多条 |
| 工作量 | `workload` | 是 | highway 页面 | 预估工作量 |

## 执行步骤

### Step 1: 收集基础信息

通过 `AskUserQuestion` 一次性收集核心信息：

```
请提供以下定制项目信息：
1. 项目名称（必填）
2. 组件名称/标识（必填）
3. JKN 单号（必填）
4. 需求评估单链接（必填）
5. 需求描述（必填，可分多条）
```

用户也可以直接粘贴 highway 需求评估单链接，从页面中提取信息。

### Step 2: 从 highway 提取信息（如提供链接）

如果用户提供了 highway 需求评估单链接，通过 browser-harness 打开页面提取信息。

#### 2.1 打开页面

```bash
browser-harness <<'PY'
url = "HIGHWAY_URL"
new_tab(url)
wait_for_load()
print(page_info())
PY
```

如果页面需要登录，提示用户手动登录后继续。

#### 2.2 先用 JS 提取页面文本（最可靠）

在截图之前，先用 JS 直接提取页面文本内容，确保不遗漏滚动区域内的信息：

```bash
browser-harness <<'PY'
import time
time.sleep(3)
# 提取页面可见文本（包含滚动区域的内容）
# 注意：js() 传入表达式即可，不要用箭头函数（箭头函数返回的是函数对象而非调用结果）
text = js("document.body.innerText")
with open("highway_text.txt", "w", encoding="utf-8") as f:
    f.write(text)
print("页面文本已保存到 highway_text.txt")
PY
```

读取 `highway_text.txt`，从中定位以下关键信息：
- **订单编号** → 这就是 JKN 单号（格式如 `JKN20260506_952`，不要从 URL 中的 RFC 编号推断）
- 项目名称
- 组件名称/标识及版本
- 需求描述/需求内容
- 工作量

#### 2.3 确认提取结果

提取信息后与用户确认：

```
从 highway 页面提取到以下信息：
- 项目名称：xxx
- 组件标识：xxx
- JKN 单号（订单编号）：xxx
- 需求描述：
  1、xxx
  2、xxx
- 工作量：xxx

请确认是否正确，或提供补充信息。
```

### Step 3: 结构化存储

将收集到的信息写入项目工作目录的 `docs/requirement.md`：

```markdown
# 定制项目需求

## 项目信息

- 项目名称：{{projectName}}
- 组件标识：{{component}}
- JKN 单号：{{jkn}}
- 需求评估单：{{requirementUrl}}
- 工作量：{{workload}}

## 需求描述

1. {{requirement_1}}
2. {{requirement_2}}
...
```

同时将关键参数输出到控制台供后续 skill 使用：

```
===CUSTOM_INFO_START===
projectName: xxx
component: xxx
jkn: xxx
requirementUrl: xxx
workload: xxx
requirements:
  1. xxx
  2. xxx
===CUSTOM_INFO_END===
```

### Step 4: 信息确认

向用户展示最终收集到的完整信息清单，确认无误后完成。用户确认后可依次触发：
1. `hido-create-version` — 使用 component、jkn、versionType 等参数
2. `custom-code-setup` — 等邮件后拉取代码
3. `custom-develop` — 使用 requirements、requirementUrl 驱动开发

## 注意事项

1. **信息来源优先级**：highway 页面文本提取（JS innerText） > 截图辅助 > 用户直接输入。highway 提取的信息需与用户确认
2. **JKN 单号来源**：JKN 单号必须从页面上的"订单编号"字段提取，**绝不能**从 URL 中的 RFC_XXX 编号推断。URL 编号是需求评估单编号，不是 JKN 工单号
3. **需求描述格式**：尽量保留原始描述，每条需求独立编号，便于后续 custom-develop 逐条实现
4. **highway 页面结构可能变化**：用 JS 提取文本，截图仅用于辅助验证
5. **不要自动跳到下一步**：信息收集完成后暂停，等用户确认再触发后续 skill

# 定制项目全流程 Skills

定制项目从信息收集到构建发布的完整工作流 skill 集合。

## 工作流程

```
信息收集 → 创建版本 → 拉取代码 → AI 开发 → 生成报告 → 正式构建
```

| 顺序 | Skill | 说明 | 关键词 |
|------|-------|------|--------|
| 1 | custom-info-collector | 收集项目信息（highway/JKN/用户输入） | "收集信息"、"定制信息"、"需求评估单" |
| 2 | hido-create-version | 在 hido 平台创建组件定制版本 | "新建版本"、"hido"、"创建版本" |
| 3 | custom-code-setup | 拉取代码，初始化开发目录 | "拉代码"、"clone"、"checkout" |
| 4 | custom-develop | AI 驱动的完整开发流程 | "开始开发"、"开发定制"、"跑开发流程" |
| 5 | test-report-generator | 生成 Excel 测试报告 | "测试报告"、"自测报告" |
| 6 | hido-build-release | 正式构建并下载成果物 | "正式构建"、"构建发布"、"打正式包" |

## 调用规则

1. **流程可跳步**：用户可以从任意步骤开始，不需要严格按顺序。例如用户直接提供代码地址时，跳到 custom-code-setup。
2. **独立可用**：custom-code-setup 和 hido-create-version 可独立使用，不依赖前置 skill。
3. **信息传递**：各 skill 之间的信息通过 `docs/requirement.md` 文件和对话上下文传递。
4. **先判断再调用**：用户提到定制相关需求时，先判断属于哪个阶段，再用 `Skill` 工具调用对应 skill。

## Skill 引用方式

调用时使用 `custompowers:<skill-name>` 格式：

- `custompowers:custom-info-collector`
- `custompowers:hido-create-version`
- `custompowers:custom-code-setup`
- `custompowers:custom-develop`
- `custompowers:test-report-generator`
- `custompowers:hido-build-release`

## 典型场景

### 场景 A：完整流程（从零开始）

用户说"有个新的定制项目" → custom-info-collector → hido-create-version → 等邮件 → custom-code-setup → custom-develop → test-report-generator → hido-build-release

### 场景 B：已有代码，直接开发

用户说"代码已经拉好了，开始开发" → custom-develop

### 场景 C：只创建版本

用户说"帮我在 hido 上建个版本" → hido-create-version

### 场景 D：只生成报告

用户说"帮我生成测试报告" → test-report-generator
